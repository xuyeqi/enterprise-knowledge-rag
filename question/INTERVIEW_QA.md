# 企业知识库 RAG 项目面试问答

## 文档定位

这份文档用于准备 20k 以下的 AI 应用开发、前端＋AI 应用开发和大模型应用开发岗位面试。

问题参考了 2026 年 7 月仍可检索到的 AI 应用开发岗位要求、RAG 面经和面试题总结，再结合本项目已经实现的功能重新组织。它们不是对某家公司原题的逐字复制，而是当前市场高频考点的项目化版本。

使用时不要逐字背诵。每道题先记住一句大白话，再按照“我做了什么→为什么这么做→现在有什么不足”往下讲。技术名词说出来以后，最好顺手用一句人话解释。

## 问题一：请介绍一下这个企业知识库 RAG 项目，你主要解决了什么问题？

### 面试官想考察什么

- 能不能在 1～2 分钟内讲清项目价值，而不是罗列技术名词。
- 是否真正做过完整项目，而不是只调用过一次大模型 API。
- 前端开发经验如何迁移到 AI 应用开发岗位。

### 参考回答

简单说，这个项目就是把公司内部的文档，做成一个可以追问、还能查看答案出处的 AI 知识库。

我做它主要有两个原因。第一个是公司制度、产品介绍、项目资料通常散落在各种 TXT、Markdown 和 PDF 里，人工查起来比较慢。第二个是直接问大模型不够可靠，它不知道公司的真实情况，很容易自己编。所以我的思路不是让模型直接回答，而是先从公司资料里找出相关内容，再让模型只根据这些内容回答。

整个过程可以理解成：先把资料整理好放进一个“能按语义搜索的资料库”，用户提问时再从里面找证据。

```text
文档上传
→ 文档解析
→ 中文切片
→ Embedding 向量化
→ PostgreSQL＋pgvector 入库
→ 用户问题向量化
→ 余弦相似度检索
→ 低相关资料过滤
→ LangChain 组织提示词
→ 百炼大模型流式回答
→ Vue 前端逐块展示答案和引用来源
```

技术上，后端用 FastAPI 接接口，LangChain 负责组织提示词和调用模型，PostgreSQL＋pgvector 存文档和向量；前端还是我比较熟悉的 Vue 3＋TypeScript。现在已经能上传文本型 PDF，保留引用页码，支持最近五轮追问。如果资料不相关，系统会拒答；如果资料相关，答案会像打字一样逐步显示。

这个项目对我最大的价值，不是证明我会调一次大模型 API，而是让我完整走了一遍 AI 功能怎么变成真实产品：文档怎么处理、数据怎么存、问题怎么检索、模型怎么约束、前后端怎么传流式数据、出错后怎么定位。

### 可能追问：为什么不直接使用 Dify 等低代码平台？

Dify 这类平台很适合快速搭 Demo，但我这个项目主要是为了转岗和面试，所以我更想把里面的关键链路亲手做一遍。这样面试官问我切片怎么来的、向量怎么存的、为什么要拒答、流式回答为什么卡住，我都能结合代码和真实问题回答，而不是只会拖工作流节点。

### 回答禁区

- 不要只说“用了 LangChain、Vue、FastAPI”。
- 不要声称项目已经达到大型企业生产规模。
- 不要把 LangChain 说成 RAG 的核心算法，它在本项目中主要负责提示词链和模型调用编排。

## 问题二：这个 RAG 系统从文档上传到生成答案，离线和在线链路分别是什么？

### 面试官想考察什么

- 是否理解 RAG 不只是“查向量库再调模型”。
- 能否区分离线索引和在线问答。
- 是否考虑事务、模型一致性和引用溯源。

### 参考回答

我会把它分成两段来讲：第一段是提前整理资料，第二段是用户提问后现场找答案。

第一段可以理解成“把资料整理上架”：

1. 接收不超过 2 MB 的 TXT、Markdown 或文本型 PDF。
2. 先检查文件类型、大小和内容，坏文件或者空文件不会继续处理。
3. 普通文本直接解码，PDF 用 `pypdf` 一页一页提取文字，同时记住每段内容来自第几页。
4. 长文档不能整篇拿去检索，所以我会尽量按段落和句子切成小块。
5. 每个小块交给 Embedding 模型，转成一个 1024 维向量。可以把向量理解成这段文字在“语义地图”里的坐标。
6. 最后把文档信息、切片文字、页码和向量一起写进 PostgreSQL。这里用了事务，也就是要么整份文档都写成功，要么全部撤销，不留半份数据。

第二段是“用户提问后现场找答案”：

1. FastAPI 先检查问题和最近五轮历史是否合法。
2. 如果用户问“它怎么样”，我会带上最近两个历史问题去检索，帮助系统知道“它”指什么。
3. 用户的问题也用同一个 Embedding 模型转成向量。
4. pgvector 比较问题向量和文档切片向量，找出语义上最接近的内容。
5. 找到以后还不能直接交给大模型。我会再过一道 `0.5` 的相似度门槛，太不相关的资料直接丢掉。
6. 有可靠资料时，LangChain 才把历史、资料和当前问题组织成提示词交给模型。
7. 模型一边生成，后端一边通过 SSE 往前端发，用户不需要等完整答案生成完。
8. 最后再把模型真正看到的引用切片发给前端，保证答案和来源能对上。

这里我最在意两个“一致”。第一，文档和问题必须用同一个 Embedding 模型，否则就像拿两张比例尺不同的地图比较坐标，距离没有意义。第二，页面展示的来源必须真的是模型看过的资料，不能拿别的切片来凑引用。

### 可能追问：为什么业务数据和向量都放 PostgreSQL？

因为这个项目现在数据量不大，我不想为了“技术栈看起来多”再单独维护一个向量数据库。PostgreSQL 本来就要保存文档状态、页码和外键，加上 pgvector 后也能做向量检索，事务还比较好处理。等以后数据真的多到查询变慢，再根据测试结果考虑 HNSW 或独立向量数据库，而不是提前把架构做复杂。

## 问题三：文档如何切片，`similarity` 又是怎么计算的？当前方案有什么问题？

### 面试官想考察什么

- 是否理解切片会直接影响召回质量。
- 是否知道向量相似度的实际计算位置。
- 能否主动说出当前方案的局限，而不是只讲优点。

### 参考回答

切片可以理解成把一本长文档拆成很多张“知识卡片”。卡片太大，里面会混进多个主题；卡片太小，又可能只剩半句话，前后意思接不上。

我现在的做法是每个切片最多 800 个字符，相邻切片尽量重复 120 个字符。切的时候不是到 800 就硬切，而是先找段落和换行，再找句号、问号、分号、逗号，实在找不到合适位置才按单个字符切。

这 120 个重叠字符相当于两张卡片之间留一点交集。即使一句话刚好落在边界附近，也更可能在下一张卡片里看到完整内容。不过它仍然只是按字符和标点判断，并不真的知道标题、表格和正文之间是什么关系。

PDF 还有一个特殊问题：我是按页解析的。好处是引用能准确显示第几页，坏处是跨页内容可能断开。比如第一页最后写“报销需要满足以下条件”，真正的条件在第二页，这两部分目前不会自动拼在一起。

检索时，问题也会被转成向量。然后 pgvector 比较问题和每张知识卡片在“语义地图”上的方向是否接近。代码里先算余弦距离，再用 `1 - 距离` 转成更容易理解的相似度：

```text
cosine_distance = embedding <=> query_vector
similarity = 1 - cosine_distance
```

所以距离越小越相关，相似度越大越相关。但 `similarity = 0.8` 不能理解成“答案有 80% 概率正确”，它只能说明问题和这段文字在语义上比较接近。

当前方案最大的三个不足是：

1. 前端目前只召回 1 个切片，可能放大跨切片语义缺失。
2. PDF 没有跨页重叠，标题和正文可能分开。
3. 固定字符切片不适合复杂表格和结构化 Markdown。

我不会看到一次回答不好就马上把 800 改成 1000。更合理的做法是先准备一批问题，看正确内容到底是解析丢了、被切断了，还是切片存在但没有召回。确认问题后，再对比召回 1～3 个切片、补充相邻切片、按标题切片或者做父子切片。

### 可能追问：为什么不直接把 `chunk_size` 调大？

因为调大不一定更好。切片太大，一张卡片里可能同时出现报销、考勤和请假三个主题，向量就很难准确表示它到底在讲什么；而且交给模型的无关文字和 Token 也会增加。切片太小又会丢上下文，所以这个参数不能拍脑袋决定，要用真实问题验证。

## 问题四：你如何减少模型幻觉？为什么相似度阈值设置为 `0.5`？

### 面试官想考察什么

- 是否区分“检索错误”和“生成错误”。
- 是否有拒答、引用和评估意识。
- 能否诚实解释经验阈值，而不是伪造精确结论。

### 参考回答

我不敢说这个项目能彻底消灭幻觉，我做的是尽量不给模型瞎编的机会。现在主要有三道关。

第一道关是先检查找到的资料到底像不像。向量数据库有个特点：哪怕用户问的问题和知识库完全没关系，它也会硬选出几个“相对最像”的切片，就像书架里没有你要的书，管理员还是递给你一本最接近的。如果把这些资料直接交给模型，它就可能顺着错误资料往下编。

所以我加了 `0.5` 的相似度门槛。低于这个值的切片不会交给模型；如果所有切片都不合格，系统就直接告诉用户“知识库里没有足够依据”，同时也不再调用大模型。这样既能减少乱答，也能省掉一次模型费用。

第二道关是告诉模型回答规则。提示词会明确要求它只能根据 `<knowledge>` 里的资料回答，聊天记录只用来理解“它”“刚才那个制度”这类指代，不能当成事实来源。即使上传的文档里写了类似“忽略前面的要求”，也只把它当普通文档内容，不让它改变系统规则。

第三道关是把依据展示出来。前端会显示这次回答实际用了哪个文件、哪个切片、PDF 第几页，以及对应正文和相似度。模型真答错了，用户和开发者都能顺着来源往回查，而不是只看到一句不知道从哪来的答案。

`0.5` 不是一个放到所有项目里都正确的标准答案，它只是我根据当前少量相关问题和无关问题定下来的初始线。更稳妥的做法，是准备一批固定测试题，例如：

- 应该命中的问题及其目标文档／页码。
- 应该拒答的知识库外问题。
- 容易混淆的近似问题。
- 连续追问和指代问题。

然后用这批题比较不同阈值、Top-K 和切片大小，看哪组参数既能找回该找的资料，又能拒绝不该回答的问题，同时还要关注答案是否正确、引用是否对得上以及响应速度。

如果系统答错，我会先看正确资料有没有被检索出来。没有找回来，问题通常在文档解析、切片、Embedding 或召回这一段；资料已经找回来了，模型还是答错，才继续检查提示词、上下文怎么拼接以及模型生成。这样能先判断是“没找到资料”，还是“找到了但没用好”。

### 可能追问：只用相似度阈值够吗？

不够，相似度阈值只能算第一道门。向量搜索擅长找“意思相近”的内容，但遇到员工编号、合同编号、产品型号这类精确词时，关键词搜索可能更可靠。后续可以把关键词搜索和向量搜索结合起来，再根据部门、文档类型等信息过滤，或者让 Rerank 模型把候选资料重新排一次顺序。

不过我不会一开始就把这些技术全堆上去。我会先收集系统答错的真实问题，看看主要是精确词搜不到、上下文断了，还是排序不准，再补对应能力。这样每个改动都有明确原因，也方便验证它到底有没有效果。

## 问题五：流式回答是如何实现的？为什么选择 SSE 而不是 WebSocket？

### 面试官想考察什么

- 是否理解模型流、HTTP 流和页面响应式更新之间的完整链路。
- 能否解释 SSE、`ReadableStream` 和网络分块。
- 是否做过真实前后端联调。

### 参考回答

本质上就是把“等模型把整段话说完再返回”，改成“模型说出一点，后端就立刻往前端发一点”。用户不用一直盯着空白页面，而是能马上看到文字逐步出现。

为了不影响原来的调用方，我保留了 `POST /answer` 这个一次性返回 JSON 的接口，另外增加 `POST /answer/stream` 专门处理流式回答。

模型调用原来用的是 LangChain 的：

```python
answer = await chain.ainvoke(input_data)
```

改为：

```python
async for content in chain.astream(input_data):
    yield content
```

`ainvoke` 要等完整答案生成后才能拿到结果，`astream` 则能一段一段拿到模型输出。后端通过 `async for` 持续接收，接到一段就用 `yield` 往前端发送一段。

不过在调用模型前，问题校验、生成 Embedding 和向量检索还是要先完成。检索不到合格资料时，后端直接发送拒答结果，不调用模型；找到资料后，才开始边生成边发送答案。

FastAPI 使用 `StreamingResponse` 返回 `text/event-stream`。整个响应不是一个完整 JSON，而是一连串小消息，每条消息仍然可以在 `data` 里放 JSON。事件顺序是：

```text
delta → delta → ... → sources → done
```

这里的 `delta` 表示“新生成的一小段答案”，`sources` 表示“这次回答引用了哪些资料”，`done` 表示“这一轮已经结束”。如果中途出错，就发送 `error`。前端根据事件名称就知道该把数据放到哪里，不需要猜它是正文还是引用。

流已经开始后，如果上游突然报错，HTTP 响应头早就发出去了，这时不能再把状态码改成 `500`。所以后端会把真实错误记进日志，只给前端发送一条不暴露内部信息的 `error` 事件。

前端使用原生 `fetch` 发送 POST 请求，再通过：

```ts
const reader = response.body?.getReader();
```

不断读取服务端后续发来的数据。`reader.read()` 拿到的还是网络字节，也就是 `Uint8Array`，所以要先用 `TextDecoder` 把它还原成字符串。

这里必须用 `buffer`，因为网络不会保证一次刚好送来一条完整事件。它可能把一条消息拆成两半，也可能一次塞进来三条消息。`buffer` 就像临时收件箱：先把每次收到的内容拼起来，遇到 SSE 规定的空行分隔符后，再拿出一条完整事件处理，剩下那半条继续等下一批数据。

前端收到 `delta`，就把文字追加到当前答案里，Vue 会自动刷新页面；收到 `sources`，就更新引用列表；收到 `done`，就关闭“正在生成”的状态。页面只展示最新一次回答，但后端还保留最近 5 轮对话，方便模型理解“刚才那条规定”“那它怎么报销”这样的连续追问。

我选择 SSE，是因为这个功能主要是服务端不断把答案推给浏览器，数据方向基本是单向的。可以把 SSE 理解成听直播：浏览器发起一次提问后，主要负责持续接收。WebSocket 更像打电话，双方都可以长期、频繁地说话，适合实时语音、在线协作或游戏。当前项目暂时不需要这种双向长连接，用 WebSocket 反而要多处理心跳、断线重连和连接状态，复杂度更高。

### 可能追问：为什么前端最初收到 `delta`，页面却没有更新？

我遇到过一个比较典型的问题：浏览器的 Network 面板里明明一直能看到 `delta`，但页面只显示开头几个字，而且“正在生成”一直不结束。

这说明后端其实已经在正常发数据，问题更可能出在前端状态更新。我继续检查后发现，代码一直在修改放进 Vue 响应式数组之前的那个普通对象，而页面真正监听的是 Vue 包装后的代理对象，所以后续修改没有稳定触发界面刷新。

修复时，我先用 `reactive()` 把当前回答变成响应式对象，再放进数组，后面收到流式内容时始终修改这个对象。

这个问题也让我形成了一套排查顺序：先看后端有没有持续发送，再看浏览器有没有持续收到，最后看收到的数据有没有真正更新到 Vue 管理的状态里。这样可以快速判断问题是在模型、网络传输，还是页面渲染。

## 面试表达检查清单

- 先讲业务问题和设计目标，再讲框架名称。
- 明确哪些功能已经实现并验证，哪些只是下一步计划。
- 不说“完全解决幻觉”，改说“通过拒答、提示词和引用降低风险”。
- 不把 `similarity` 解释成答案正确概率。
- 不声称 `0.5` 是通用最优阈值。
- 不声称当前已经支持十万级文档；可以说明大规模场景需要异步索引、近似索引、增量更新和性能评估。
- 遇到追问时按照“现状→原因→不足→验证后的优化方向”回答。
